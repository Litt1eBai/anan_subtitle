# RELEASE

本文档描述当前项目的发布建议、打标签方式，以及正式版前需要补齐的事项。

## 当前建议的发布级别

按当前代码库状态，更适合发布：

- `v0.x` 预览版
- `beta` 或 `preview` 小范围内测版

当前还不建议直接作为“正式稳定版”发布给普通用户，原因包括：

- 还没有真正的 Windows `exe` 打包方案
- 自动化测试主要是单元测试，缺少端到端验证
- 依赖版本还没有收敛
- 首次启动和模型下载路径还需要按最终产品体验再打磨

## 是否可以只用 Git 标签发布

可以，但要区分“源码版本发布”和“面向普通用户的正式交付”。

当前项目如果只是：

- 标记一个可回溯版本
- 给内测用户或开发者一个固定源码快照
- 配合发布说明或 release note

那么使用 Git 标签就足够了。

但如果目标是：

- 给普通用户一个可直接双击使用的版本
- 提供稳定可重复安装的 Windows 交付物

那么 Git 标签本身不够，还需要配套：

- Windows `exe` 构建产物
- 构建脚本或 spec
- 发布说明
- 冒烟验证记录

## 当前推荐的发布流程

### 1. 先整理工作区

- 确保 `git status` 干净
- 不要把本机运行态配置直接带进发布提交
- 确保当前提交已经通过测试

当前仓库里 `config/app.yaml` 属于运行态配置，不应直接作为发布基线的一部分频繁变动。

### 2. 跑发布前检查

至少执行：

```powershell
$env:PYTHONPATH='src'
.\.venv\Scripts\python.exe -m unittest discover -s tests -v
.\.venv\Scripts\python.exe -m compileall src
```

如果后续补了打包流程，还应加入：

- `exe` 构建验证
- 安装后启动验证

### 3. 准备发布说明

建议至少写清楚：

- 版本号
- 本次主要功能
- 已知限制
- 适用平台
- 首次运行是否会下载模型

### 4. 创建注释标签

推荐使用注释标签，而不是轻量标签。

示例：

```powershell
git tag -a v0.1.0-beta.1 -m "v0.1.0-beta.1"
```

如果这是正式版：

```powershell
git tag -a v1.0.0 -m "v1.0.0"
```

### 5. 推送提交和标签

```powershell
git push origin main
git push origin v0.1.0-beta.1
```

如果远端平台支持 release 页面，再基于该标签创建 release。

## 版本命名建议

当前阶段建议使用预发布版本号：

- `v0.1.0-preview.1`
- `v0.1.0-beta.1`
- `v0.1.0-beta.2`

等到下面的正式版前清单补齐后，再考虑：

- `v1.0.0`

## 正式版前至少要补的东西

### 1. 真正的 Windows 打包方案

至少补齐其一：

- `exe` 构建脚本
- PyInstaller spec
- 明确的发布产物目录和构建命令
- `PySide6` 相关动态链接分发方案与 `LGPL` 声明附带方式

目标是让发布不再依赖用户本机自己建虚拟环境和安装依赖。

### 2. 一次人工冒烟清单

至少覆盖这些路径：

- 启动应用
- 切换识别模式
- 托盘显示/隐藏
- 保存配置
- 退出后重启

建议将每次发布前的冒烟结果记录下来。

### 3. 至少一条端到端验证路径

当前测试主要保护重构，不足以替代真实发布验证。

正式版前至少应补一条端到端路径，例如：

- 启动应用
- 加载模型
- 打开麦克风
- 输出字幕
- 退出并回收资源
- 验证模型来源说明与发布包中的许可证文件齐全

可以是人工验证，也可以逐步演进成自动化脚本。

### 4. 依赖版本收敛

当前 `requirements.txt` 使用的是宽版本范围。

正式版前建议至少做到：

- 固定核心依赖版本
- 避免每次安装得到不同结果
- 明确兼容的 Python 版本

### 5. 模型来源与使用条款说明

正式发布前应对实际使用到的模型逐个补充来源说明，至少包含：

- 模型名称
- 模型来源平台或仓库
- 对应链接
- 相关使用条款或许可说明

如果发布包会自动下载模型，也应在 README 或发布说明中写清这些模型来源。

### 6. 清理本机运行态配置，做一个干净 tag

正式发布前应保证：

- 工作区干净
- 发布提交不混入本机临时状态
- 标签对应一个可复现的干净提交

尤其是：

- `config/app.yaml` 这类运行态配置要谨慎处理

## 一句话建议

当前阶段可以通过 Git 标签发布一个 `v0.x beta/preview` 源码版本；正式版发布则应建立 `exe` 构建、冒烟清单、端到端验证、依赖收敛和干净标签这几项基础能力。
